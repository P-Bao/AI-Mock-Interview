# 1. Mục tiêu & Giả thuyết MVP
## 1.1 Giả thuyết cốt lõi
*“Sinh viên CNTT sẽ cảm nhận buổi phỏng vấn giả lập hoàn toàn bằng giọng nói là đủ thực tế và tạo ra áp lực — chỉ nhờ vào phương thức giọng nói — mà không cần camera hay người phỏng vấn thật.”*

## 1.2 Mục tiêu học hỏi (theo thứ tự ưu tiên)

| **Thứ tự** | **Mục tiêu học hỏi** | **Cách đo lường** |
| --- | --- | --- |
| 1 (Chính) | Pipeline Voice→STT→AI→TTS hoạt động ổn định, liền mạch trong điều kiện thực tế | Latency < 5s, error rate < 10%, không có crash trong session hoàn chỉnh |
| 2 | Sinh viên thực sự cảm thấy áp lực và tính thực tế qua voice-only | ≥ 60% sinh viên đánh giá “thực tế” hoặc “gây áp lực” trong khảo sát sau session |
| 3 | Phản hồi STAR đơn giản đủ giá trị để sinh viên muốn luyện lại | ≥ 40% sinh viên thực hiện ≥ 2 session trong tuần thử nghiệm |



# 2. Đối tượng mục tiêu (MVP)

| **Thuộc tính** | **Mô tả** |

# 3. Tập tính năng cốt lõi (IN vs OUT)
## 3.1 Tính năng IN — Bắt buộc cho MVP

| **Tính năng** | **Tham chiếu PRD** | **Lý do bắt buộc** |
| --- | --- | --- |
| Bắt đầu không cần đăng nhập / tạo tài khoản | US-01 | Giảm ma sát tối đa cho cohort lớp học |
| Yêu cầu quyền microphone + thông báo lỗi rõ ràng | US-03 | Pipeline voice không thể hoạt động nếu thiếu bước này |
| AI đọc câu hỏi tiếng Việt qua TTS | US-04 | Cốt lõi giả thuyết — phương thức giọng nói tạo ra áp lực |
| STT real-time + hiển thị transcript | US-05 | Cho phép xác nhận hệ thống nghe đúng câu trả lời |
| Audio visualizer (sóng âm) khi ghi âm | US-06 | Phản hồi thị giác — người dùng biết mic đang hoạt động |
| Hiển thị trạng thái AI rõ ràng (Đang hỏi / Nghe / Phân tích) | US-07 | Tạo cảm giác hội thoại tự nhiên — quan trọng cho trải nghiệm |
| Bộ đếm thời gian toàn session | US-08 | Góp phần tạo áp lực thời gian thực tế |
| Câu hỏi được chọn ngẫu nhiên / động | Mục 6.1 PRD | Tăng tính thực tế, sinh viên không chuẩn bị trước theo kịch bản |
| Kết thúc session sớm bất kỳ lúc nào | US-11 | Kiểm soát người dùng — UX cơ bản |
| Sau session: Tổng điểm + STAR breakdown + 1 gợi ý/câu | US-12, 13, 16 | Kiểm tra giá trị phản hồi đủ để quay lại |
| Lưu lịch sử session (local storage) — thời gian + điểm | US-17 | Theo dõi tiến bộ cơ bản không cần backend/đăng nhập |


## 3.2 Tính năng OUT — Loại khỏi MVP

# 4. Technology Stack
## 4.1 Quyết định API (Spike Sprint — Tuần 1)
*API voice là rủi ro kỹ thuật số 1. Tuần 1 dành hoàn toàn để PoC và quyết định.*


| **Phương án** | **Ưu điểm** | **Rủi ro** | **Khuyến nghị** |
| --- | --- | --- | --- |
| Gemini Live API (end-to-end) | All-in-one, latency thấp, context dài | Beta/unstable, tiếng Việt chưa kiểm chứng, khó debug | Thử nghiệm trong Spike Sprint trước |
| Google STT + Gemini Text + Google TTS (tách riêng) | Mature, dễ debug từng bước, đo latency chính xác | 3 API calls → latency cao hơn | Fallback nếu Gemini Live không ổn định |


## 4.2 Stack đề xuất (chờ xác nhận sau Spike Sprint)

| **Layer** | **Lựa chọn** | **Lý do / Giới hạn free tier** |
| --- | --- | --- |
| STT | Gemini Live API hoặc Google STT v2 (streaming) | Tiếng Việt tốt nhất; Google STT free: 60 phút/tháng |
| AI Evaluation | Gemini 1.5 Flash (text) | Free tier: 15 RPM, 1M tokens/ngày — đủ cho 80 SV |
| TTS | Google Cloud TTS (Neural2 vi-VN) hoặc Gemini Live TTS | Free: 1 triệu ký tự/tháng (Standard), 100K (Neural2) |
| Frontend | React + Vite | FE specialist đã có; Web Audio API cho visualizer |
| Backend | Python + FastAPI | BE specialist đã có; async tốt cho streaming audio |
| Database | Local Storage (không cần backend DB cho MVP) | Không cần đăng nhập — đơn giản hóa toàn bộ stack |
| Hosting FE | Vercel (free tier) | Deploy nhanh, HTTPS mặc định, CDN |
| Hosting BE | Railway hoặc Render (free tier) | Đủ cho 80 SV; có thể ngủ — cần warm-up cho demo |


## 4.3 Lưu ý đặc biệt về free tier
1. Gemini 1.5 Flash: 15 requests/phút — đủ cho từng sinh viên dùng cá nhân, cần rate limiting nếu mọ cùng lúc
1. Google STT: 60 phút audio/tháng (miễn phí) — ước tính: 80 SV x 20 phút = 1,600 phút → cần kiểm tra giới hạn hoặc dùng key ring
1. Railway/Render free tier: server có thể “ngủ” sau 15 phút không dùng — phải có warm-up script trước buổi demo hội đồng

# 6. Chiến lược Kiểm thử
## 6.1 Kiểm thử kỹ thuật (liên tục từ Tuần 1)

| **Loại test** | **Nội dung** | **Thời điểm** |
| --- | --- | --- |
